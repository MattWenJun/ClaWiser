# HDD: Hypothesis-Driven Development

六步流程：**HT → PT → AT Design → DEV → AT Run → E2E**

每个测试阶段的核心动作相同：**写测试用例 → 执行 → 判定通过/不通过**。
信息收集（查数据、搜网、读文档、跑命令）是写测试用例的输入，不是判定依据。

## 流程

### 1. HT: Hypothesis Test（假设验证）

提出假设，用测试用例验证。

```
- 假设: [对问题根因的判断]
- 信息收集: [查本地文件/搜网/跑命令 — 为写测试服务]
- 测试用例:
  - HT1: [验证条件] → 预期: [xxx] → 实际: [xxx] → ✅/❌
  - HT2: ...
- 判定: ✅ 进入 PT / ❌ 替换假设，重跑 HT
```

**关键规则：**
- 不能提了假设就一直绕，必须先用测试用例确认假设是否成立
- 假设不通过 → 必须替换假设，不能在错误假设上调方案

**假设提出后必须自我攻击：**
1. **攻击 Ground Truth**：我的判断依据本身有没有破绽？有没有反例？
2. **检查其他可解释性**：同一现象是否有完全不同的解释？（问题可能不是"怎么做 X"而是"需不需要 X"）
3. **找反例**：已有事实中，有没有直接反驳假设的证据？

三项检查不通过 → 假设不成立，重新提假设。

### 2. PT: Plan Test（方案验证）

提出方案，用测试用例验证可行性。

```
- 方案: [具体方案描述]
- 信息收集: [查文档/搜网/确认可行性 — 为写测试服务]
- 测试用例:
  - PT1: [方案前提是否成立] → 预期: [xxx] → 实际: [xxx] → ✅/❌
  - PT2: [关键参数是否可用] → 预期: [xxx] → 实际: [xxx] → ✅/❌
- 判定: ✅ 进入 AT Design / ❌ 调整方案 / ❌❌ 回退 HT
```

**回退规则：**
- PT 失败但假设没问题 → 调整方案，重跑 PT
- PT 失败且发现假设有问题 → 回退 HT，重新提假设

### 3. AT Design: Acceptance Test Design（验收用例编写）

在写代码之前，先定义"做完长什么样"。

```
- 验收标准:
  - AT1: [输入/操作] → 预期结果: [xxx]
  - AT2: [边界条件] → 预期结果: [xxx]
  - AT3: ...
```

**关键规则：**
- 必须在 DEV 之前完成
- 用例要具体可执行，不能是模糊描述

### 4. DEV: Development（开发）

基于 PT 方案 + AT 用例执行开发。

```
- 改动清单:
  - [文件1]: [改了什么]
  - [文件2]: [改了什么]
```

### 5. AT Run: Acceptance Test Run（验收测试执行）

执行 AT Design 中定义的用例。

```
- AT1: → 实际: [xxx] → ✅/❌
- AT2: → 实际: [xxx] → ✅/❌
- 判定: ✅ 进入 E2E / ❌ 回 DEV 修复
```

### 6. E2E: End-to-End Test（端到端验收）

由用户亲自验收。

```
- E1: [验收项] → ✅/❌
- E2: ...
- 判定: ✅ 完成 / ❌ 回退 DEV 或 PT
```

## 回退路径

```
HT ←────────────────────┐
 ↓ pass                  │
PT ←─────┐               │
 ↓ pass  │ PT fail       │ 根因在假设
AT Design│               │
 ↓       └→ 调整方案 ────┘
DEV
 ↓
AT Run ──→ fail → 回 DEV
 ↓ pass
E2E ────→ fail → 回 DEV 或 PT
 ↓ pass
Done ✅
```

### 6.1 收尾检查（E2E 通过后必做）

E2E 通过后，执行以下 checklist：

```
- [ ] 是否需要新建/更新项目文档？（判断标准见 AGENTS.md）
- [ ] 是否有新的经验教训需要记录？
- [ ] 相关的 memory/YYYY-MM-DD.md 是否已更新？
```

不满足任何一项可以跳过，但必须**逐项过一遍**再标记 Done。

## 何时使用

**需要 HDD（满足任一）：**
- 涉及"为什么"（诊断/debug）
- 涉及"哪种方案更好"（优化/选型）
- 改动影响上下游（架构/配置变更）
- 结果不确定，需要验证

**不需要 HDD：**
- 指令明确、结果确定（"把标题改成xxx"）
- 纯信息查询（"这个 API 怎么用"）
- 简单追加/删除操作

## 信息收集指引

HT 和 PT 阶段都可能需要信息收集，作为写测试用例的输入：

- **查本地文件**: `read`/`exec` 读配置、日志、源码
- **搜向量库**: `memory_search` 查历史决策和上下文
- **搜网**: `grok_search`/`web_fetch` 查文档、方案、已知问题
- **跑命令**: `exec` 验证环境、版本、状态

信息收集不是目的，写出可执行的测试用例才是。

## 异常模式识别（Anomaly Detection）

HT 阶段之前，还有一个更根本的能力：**从数据中发现异常**。很多 bug 不是用户报告的，而是在做别的事时偶然注意到的。

### 看整体模式，不只看单条数据

单条结果对不对很难判断，但整体分布的异常一眼就能看出：

- **不自然的一致性**：一个分类器把所有输入都归为同一类 → 输入可能是错的（分类逻辑错会随机错，但全部一样说明看到的是同一种东西）
- **不该出现的空值/默认值**：某个字段全是 null 或 default → 上游数据没传进来
- **计数不对**：预期 8 种分类结果但只看到 1 种 → 系统性问题

**实战案例（某规则分类系统，2026-02-26）：**
原本在讨论"怎么新增分类"，看 debug log 时发现所有消息（包括 `"good"`、`"可以推进"`、所有语音）全部被分为 `recall`。分类器不可能把所有消息都归同一类——除非它根本没看到实际内容。这个观察直接改变了问题方向：从"加什么分类"变成"为什么分类器输入是错的"。

### 保持对"正常"的校准

要发现异常，首先要知道什么是正常。在系统运行初期：
1. 主动检查几条典型输入的处理结果，建立基线认知
2. 对关键环节保留日志/统计，方便后续发现偏移
3. 定期抽查，不只是出了问题才看日志

## 调试技术（Debug Techniques）

HT/PT 阶段经常需要调试来验证假设。以下方法经过实战验证（某规则分类系统 语音提取 bug，2026-02-26）。

### 逐步打桩法（Step-by-Step Instrumentation）

当多步处理链输出不对时：

1. 在每步之间插入 debug log（记录该步的输入和输出）
2. 对比每步结果，定位第一个"输入对但输出错"的步骤
3. 问题就在那一步的逻辑里

```javascript
// 示例：在 extractUserText 每步加 debug
dbg(`after step1: "${cleaned.slice(0, 120)}"`);
dbg(`after step2: "${cleaned.slice(0, 120)}"`);
// → 发现 step3 输入正确但输出没变 → 问题在 step3 的 regex
```

**关键：** 不要猜"可能是这步"然后直接改代码。打桩定位比猜测修复快得多，而且不会引入新问题。

### 模拟 vs 真实验证

- 本地模拟（`node -e` / 单元测试）可以快速验证逻辑
- 但**模拟的输入格式可能与真实环境不同**
- **模拟通过 ≠ 真实通过**，必须在真实环境中再验证一次
- 真实与模拟结果不一致时 → 问题在输入格式假设上

**实战教训：** 本地模拟 `extractUserText` 时用了多行文本（手动加了换行），regex 匹配成功；但真实 `event.prompt` 是单行格式（没有换行），regex 中的 `.*?\n` 匹配失败。模拟给了"代码没问题"的假象。

### 格式假设检查

正则表达式和解析逻辑依赖对输入格式的假设（换行符、分隔符、字段顺序等）。当"逻辑对但不工作"时：

1. 用 `JSON.stringify()` 打出原始文本 → 暴露不可见字符、换行、编码
2. 用 `.length` 确认文本完整性 → 排除截断
3. 用 `.startsWith()` / `.includes()` 做简单断言 → 缩小范围
4. 特别注意：**单行 vs 多行**、`\n` vs `\r\n`、空格 vs tab

### 减少调试来回

真实环境测试往往需要重启服务 + 用户配合，周期长。为减少来回：

1. **每次重启前多加几个 debug 点**，一次收集尽可能多的信息
2. **同时打印输入和输出**，不要只打输出（看不出输入是否符合预期）
3. **打印长度和关键特征**（startsWith、includes），不只是截断内容
4. **保留 debug log 到问题完全解决**，过早清理可能需要重新加回来
