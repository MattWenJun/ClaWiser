# 异常模式识别（Anomaly Detection）

HT 阶段之前，还有一个更根本的能力：**从数据中发现异常**。很多 bug 不是用户报告的，而是在做别的事时偶然注意到的。

## 看整体模式，不只看单条数据

单条结果对不对很难判断，但整体分布的异常一眼就能看出：

- **不自然的一致性**：一个分类器把所有输入都归为同一类 → 输入可能是错的（分类逻辑错会随机错，但全部一样说明看到的是同一种东西）
- **不该出现的空值/默认值**：某个字段全是 null 或 default → 上游数据没传进来
- **计数不对**：预期 8 种分类结果但只看到 1 种 → 系统性问题

**实战案例：**
在一个消息分类系统中，观察 debug log 时发现所有消息（包括 `"good"`、`"可以推进"`、语音消息）全部被分为同一类别。分类器不可能把所有消息都归同一类——除非它根本没看到实际内容。这个观察直接改变了问题方向：从"加什么分类"变成"为什么分类器输入是错的"。

## 保持对"正常"的校准

要发现异常，首先要知道什么是正常。在系统运行初期：
1. 主动检查几条典型输入的处理结果，建立基线认知
2. 对关键环节保留日志/统计，方便后续发现偏移
3. 定期抽查，不只是出了问题才看日志
