# Product First — 产品目标先于技术方案

**先回答"用户每天怎么用"，再回答"技术怎么实现"。**

常见错误：收到项目需求 → 直接做技术架构 / 开发计划 / 选型调研。
正确顺序：收到项目需求 → **产品目标 → 用户场景 → 判断依据** → 然后才是技术。

## 产品计划三要素

### 1. 用户场景（必须先写）

用"一天的使用故事"描述，不用功能列表：

```markdown
## 使用场景

### 场景 1：每日推送
用户早上打开消息，看到今天的精选内容。
5 条带一句话理由，点开链接直达原文。
不需要主动操作，推送自动到。

### 场景 2：深度展开
看到一条感兴趣的，回复"展开第 3 条"。
收到核心段落 + 相关引用 + 可能的 idea 候选。
```

**检验标准**：读完场景描述，一个不了解项目的人能画出产品原型吗？能 → 够了。不能 → 写得不够具体。

### 2. 判断依据（什么算"好"）

每个场景定义**可验证的质量标准**：

```markdown
## 判断依据

| 维度 | 标准 | 怎么验 |
|------|------|--------|
| 相关性 | 推送内容与兴趣匹配度 ≥70% | 用户反馈 |
| 时效性 | 80% 是最近 3 天的 | 查日期分布 |
| 可操作 | 每条都能一键展开 | 功能测试 |
```

### 3. Phase 划分（怎么分步做到）

基于场景**反推** Phase：
- Phase 1 = 最小可用（覆盖场景 1）
- Phase 2 = 核心体验（覆盖场景 2）
- Phase 3+ = 增强和优化

**不是按技术模块拆 Phase，是按用户价值拆。**

## 常见陷阱

| 陷阱 | 表现 | 纠正 |
|------|------|------|
| 技术驱动 | "先做抓取，再做评分，再做推送" | 倒过来：推送什么样 → 需要什么评分 → 需要什么数据 |
| 功能堆砌 | Phase 1 就塞满功能 | 砍到只剩场景 1 能跑通的最小集 |
| 跳过场景 | 直接写 Task 列表 | 先写"用户每天看到什么" |
| 假设未验 | "用 XX API 做全文搜索" → 没查过能不能用 | 每个关键假设走 HDD 验证 |

## 产出物

完成 product-first 后，应产生：

1. **product-plan.md**（存 `memory/projects/<project>/`）
   - 用户场景（至少 2 个核心场景）
   - 判断依据（每个场景的质量标准）
   - Phase 划分（按用户价值排序）
2. **Phase 1 详细计划**
   - Task 列表 + 每个 Task 的验收标准
   - 依赖图 + 阻塞项

技术选型、架构设计在产品计划确认后再做。
